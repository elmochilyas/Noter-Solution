## Bugs Found & Fixed

### BUG-014 (P1) — triageSuggestions returns raw slugs, view expects translated labels
- **File:** `app/Domain/Services/Chatbot/ChatbotService.php`
- **Problem:** `triageSuggestions()` returned raw slugs (`family`, `real_estate`, etc.) but the Blade view displayed them directly.
- **Fix:** Kept raw slugs at the service layer (correct — the triage state machine needs them). Updated `resources/views/livewire/chatbot.blade.php` to translate via `__('chatbot.category_'.$suggestion)` with fallback.
- **Test:** Replaced old assertion `'category chips show translated labels not raw slugs'` with `'triage chips send raw slugs from service suggestions'`.

### BUG-015 (P2) — getMonthlyCost sums tokens_in from user messages instead of assistant
- **File:** `app/Domain/Services/Chatbot/ChatbotService.php`
- **Problem:** `getMonthlyCost()` queried `role = 'user'` for `tokens_in`, but `LlmResponse` (which carries `tokensIn`) is only stored on assistant messages.
- **Fix:** Changed role filter to `'assistant'`.
- **Test:** Added `'getMonthlyCost correctly calculates input and output costs'`.

### BUG-016 (P2) — FAQ retrieval `orWhere` ungrouped, leaks unpublished FAQs
- **File:** `app/Domain/Services/Chatbot/ChatbotService.php`
- **Problem:** The `->where('is_published', true)` was not grouped with the `orWhere('answer_translations->...', 'LIKE', ...)`, so an unpublished FAQ could match via answer.
- **Fix:** Wrapped the `orWhere` inside a closure with `where(function ($q) { ... })`.
- **Test:** Added `'retrieveFaqs does not return unpublished FAQs'`.

### BUG-017 (P2) — buildFaqContext ignores locale parameter, uses app()->getLocale()
- **File:** `app/Domain/Services/Chatbot/ChatbotService.php`
- **Problem:** `buildFaqContext()` had no `$locale` parameter and read `app()->getLocale()` instead.
- **Fix:** Added `string $locale` parameter and used it for all translation access.
- **Test:** Added `'buildFaqContext uses the provided locale not app locale'`.

### BUG-018 (P3) — getAllowedAmounts returns null prices without filtering
- **File:** `app/Domain/Services/Chatbot/OutputFilter.php`
- **Problem:** Plans with null `price` were included in allowed amounts, leading to bad output filter comparisons.
- **Fix:** Added `->filter(fn ($v) => $v !== null)`.
- **Test:** Covered by broader `OutputFilter` tests (in existing suite).

### BUG-019 (P3) — missing parentheses in boolean condition
- **File:** `app/Domain/Services/Chatbot/ChatbotService.php`  
  (within `generateStructuredResponse`)
- **Problem:** `$budget > 0 && $this->getMonthlyCost() / $budget >= 0.8` evaluated left-to-right instead of grouping the division.
- **Fix:** Added parentheses: `$budget > 0 && ($this->getMonthlyCost() / $budget) >= 0.8`.
- **Test:** Existing budget tests already covered the correct behavior after fix.

### BUG-020 (P3) — TOCTOU race in Livewire rate limiting
- **File:** `app/Livewire/Chatbot.php`
- **Problem:** Check-then-act gap between `isBudgetExhausted()` and `$this->llm->complete()`.
- **Status:** **Deferred.** Fix complexity (distributed lock, queue, etc.) exceeds severity for v1.

### BUG-021 (P1) — IntentClassifier `containsWord` false positive on short substrings  
*(Discovered while debugging triage flow)*
- **File:** `app/Domain/Services/Chatbot/IntentClassifier.php`
- **Problem:** `containsWord()` used `str_contains($haystack, $needle)` as a fallback, which matched `'hi'` inside `'this_week'`, causing the classifier to return `GREETING` instead of `FAQ_QUERY`. This made the triage block unreachable during the urgency step.
- **Fix:** Changed `containsWord()` to use `str_contains` only for multi-word keywords (which cannot accidentally match as substrings of unrelated text). Single-word keywords use `preg_match` with `\b` prefix, correctly matching word boundaries.
- **Test:** Added `'respondTo processes triage with direct metadata'`.
