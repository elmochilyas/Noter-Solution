# Feature: Internationalization

Standards in `STANDARDS/accessibility-i18n.md`. This doc covers the user-facing feature, the editor workflow for translations, and the implementation glue.

## Supported locales

- **Arabic (`ar`)** — primary, default for new visitors from Morocco
- **French (`fr`)** — secondary

## Language detection (visitor's first arrival)

Priority order:
1. URL prefix (`/ar/` or `/fr/`) — explicit
2. `locale` cookie (set on previous visit)
3. Browser `Accept-Language` header
4. Default: `ar`

A visitor landing on the root `/` is 301-redirected to the resolved locale path.

## URL structure

- Every page: `/{locale}/{path}` where locale ∈ `{ar, fr}`.
- Home pages: `/ar/`, `/fr/`.
- Root `/` redirects.

```php
Route::get('/', LocaleRedirectController::class);
Route::prefix('{locale}')->where(['locale' => 'ar|fr'])->middleware('locale')->group(function () {
    // all locale-prefixed routes
});
```

`locale` middleware:
- Validates the locale param.
- Sets `app()->setLocale($locale)`.
- Sets `Carbon::setLocale($locale)`.
- Sets `<html lang>` and `<html dir>` in the layout.
- Updates the `locale` cookie if it differs.

## Language switcher

Visible in the header on every page. Two buttons:
- "العربية" (when current = `fr`)
- "Français" (when current = `ar`)

Click:
1. Computes the equivalent URL for the other locale.
2. Updates the cookie.
3. Navigates.

Equivalent URL computation: same path, swap the locale segment. Slugs are the **same** between locales (we don't translate slugs) to keep this trivial:
- `/ar/services/famille` ↔ `/fr/services/famille`
- `/ar/consultation` ↔ `/fr/consultation`

Exception: `politique-confidentialite` (FR) ↔ `سياسة-الخصوصية` (AR). For these, the route registers both slugs and provides a manual mapping.

## RTL handling

Arabic pages render RTL. Done via:

1. `<html dir="rtl" lang="ar">` set in the layout based on current locale.
2. Tailwind `rtl:` variant + logical properties (`ms-`, `me-`, `ps-`, `pe-`, `start-`, `end-`, `text-start`, `text-end`).
3. `tailwindcss-rtl` plugin to auto-flip directional utilities.
4. Icons with directional meaning (arrows, chevrons) get `rtl:scale-x-[-1]` or a language-specific variant.
5. SVG illustrations with directional content reviewed manually.

LTR-locked elements (phone numbers, emails, URLs, monetary amounts in Latin numerals) wrapped in `<bdi dir="ltr">`.

## Fonts per language

| Use | Latin (FR) | Arabic (AR) |
|---|---|---|
| Display headings | Fraunces (variable) | Reem Kufi (variable) |
| Body / UI | Inter (variable) | IBM Plex Sans Arabic |

Loaded via `@font-face` with `font-display: swap`. Subset by language (Latin Extended for FR; Arabic + basic Latin for AR).

Preload only the critical weight of the current language.

## Database-stored translations

CMS-managed content uses `spatie/laravel-translatable` with `jsonb` columns:

```php
class ConsultationPlan extends Model
{
    use HasTranslations;

    public array $translatable = ['name', 'description', 'features'];

    protected $casts = [
        'name' => 'array',
        'description' => 'array',
        'features' => 'array',
    ];
}
```

Stored as:
```json
{
  "ar": "استشارة عادية عبر الإنترنت",
  "fr": "Consultation standard en ligne"
}
```

Filament forms use a custom `TranslatableTabs` component to render per-locale inputs.

## What's translated

| Content | Mechanism |
|---|---|
| UI labels, buttons, errors | `resources/lang/{locale}/*.php` |
| Service pages (titles, body) | DB via spatie/laravel-translatable |
| Consultation plans | DB |
| FAQ Q&A | DB |
| Email/SMS/WhatsApp templates | `resources/lang/{locale}/notifications.php` |
| Legal pages | DB |
| Chatbot system prompt | `resources/lang/{locale}/chatbot.php` |
| Settings labels | Hardcoded in Filament (FR primarily — admin is French-speaking) |

## What's NOT translated

- Filament admin UI (French only — admins use French)
- Pulse dashboard
- Stripe checkout iframe (Stripe handles its own i18n; we pass `locale=auto`)
- Provider-rendered content (Jitsi UI)
- Logs and Sentry events

## Slugs

Service slugs in URLs are kept in **French** (the more URL-friendly script) across both locales:

- `/ar/services/famille` (URL has French slug, but content is Arabic)
- `/fr/services/famille`

This keeps URLs simple, recognizable, and shareable across language groups. Arabic-only slugs would break copy-paste for many systems.

Exception: legal pages where Sana wants the full Arabic URL — handled via custom route map.

## Numbers and dates

Per `STANDARDS/accessibility-i18n.md`:
- All numbers in Latin numerals (1, 2, 3) including Arabic pages — clarity for legal/fiscal context.
- Dates in long format with locale-appropriate month names: "14 mars 2026" / "14 مارس 2026".
- Time in 24-hour format: "10:30".
- Currency: French → "250,00 MAD"; Arabic → "250.00 درهم".

## Pluralization

Use Laravel's `trans_choice()`. Arabic has 6 plural forms — translation entries must include all forms.

```php
'slots_remaining' => '{0}لا توجد مقاعد متبقية|{1}مقعد واحد متبقي|{2}مقعدان متبقيان|[3,10]:count مقاعد متبقية|[11,*]:count مقعدًا متبقيًا',
```

## hreflang and SEO

Every page emits:

```html
<link rel="alternate" hreflang="ar" href="https://sana-bouhamidi.ma/ar/services/famille" />
<link rel="alternate" hreflang="fr" href="https://sana-bouhamidi.ma/fr/services/famille" />
<link rel="alternate" hreflang="x-default" href="https://sana-bouhamidi.ma/ar/services/famille" />
```

Plus a canonical pointing to itself.

Sitemap includes both versions of every page.

## Mixed-language users

Some users will switch back and forth — e.g. reading FAQ in French but receiving emails in Arabic. The `clients.preferred_locale` is set:
- On first booking (from the current locale at booking time)
- Updatable in portal preferences

The preferred_locale governs which language emails arrive in, regardless of the website language at the moment.

## Form input behavior

- Text inputs receive content in the user's chosen language; we don't restrict.
- Validation error messages translated.
- Placeholder examples per locale.
- Date/time inputs: native; the browser uses its locale (acceptable trade-off — most Moroccan browsers are French- or Arabic-localized).

## Translation workflow

For static strings:
1. Developer adds the FR key first.
2. Marks AR with `// TODO: translate` if temporary.
3. CI fails if keys don't parity between FR and AR.
4. Translation reviewed by native speaker before merge.

For CMS content:
1. Sana edits in Filament with the per-locale tabs visible.
2. Both locales required before publishing (validation rule on the form).
3. Preview button to see public rendering before saving.

## Native review

Before launch:
- Every Arabic translation reviewed by a native speaker (ideally Sana herself or a colleague — terminology accuracy is critical given the legal context).
- Reviewer signs off via a tracked checklist.
- Post-launch, ongoing review when content is added.

## Edge cases

| Case | Behavior |
|---|---|
| URL with unknown locale (e.g. `/en/`) | 404 (we only support ar/fr) |
| Missing translation key | Falls back to FR if AR missing; logs a warning. In dev mode shows the key prominently. |
| Email language for new client | Uses the current locale at booking time |
| Email language after preference set | Uses `client.preferred_locale` |
| Mid-conversation chatbot language switch | Detected after 2 user messages; system prompt re-rendered |
| RTL UI bug | Reported via thumbs-down in Filament; fix in next sprint |

## Acceptance criteria

- [ ] Both locales available for every public page
- [ ] Both locales available for every portal page
- [ ] Language switcher preserves current path
- [ ] Cookie persists choice
- [ ] hreflang tags present on every page
- [ ] sitemap.xml lists both versions
- [ ] RTL renders correctly on Arabic pages (visual review)
- [ ] All directional CSS uses logical properties
- [ ] All static UI strings come from translation files (no hardcoded text in PHP/Blade)
- [ ] FR and AR translation files have identical keys (verified by CI)
- [ ] CMS content forces both translations before publish
- [ ] Notification templates render correctly in both languages
- [ ] Pluralization works in Arabic for at least 3 different counts
- [ ] Dates and currency format per locale
- [ ] Mixed-direction content (Latin numbers in Arabic text) renders correctly
- [ ] Fonts loaded with correct subsets per language
- [ ] No mojibake / glyph fallback issues on common browsers
- [ ] Native speaker has signed off on all Arabic copy
- [ ] No console errors on either locale
- [ ] Lighthouse Accessibility = 100 on both languages

## Risks

- **Translation drift** — content updates in one language not mirrored. Mitigated by required-both validation in Filament + key parity check in CI.
- **RTL visual issues** — sporadic bugs at edge cases. Mitigated by visual regression tests on key pages.
- **Arabic typography quality** — Reem Kufi / IBM Plex Sans Arabic are good choices but render best on modern browsers. Old browsers may fall back to system Arabic fonts (still readable).
- **Native review bottleneck** — depends on Sana's availability. Schedule review explicitly in the phase plan.
